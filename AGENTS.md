# リポジトリガイドライン

## プロジェクト構成とモジュール整理
このリポジトリは、LP（ランディングページ）計画用のドキュメント中心の作業スペースです。内容の大半はMarkdownのメモと構成資料で、1つのSVGアセットがあります。

- ルート: 参考メモやリサーチ資料のMarkdown（例: `Every Layout.md`, `Refactoring UI.md`）。
- `LP設計案/`: 番号付きの設計ドキュメント（`01_...`〜`04_...`）とフロー図`03_再設計フロー.svg`が揃った中核フォルダ。

現時点ではアプリケーションのソースツリーやテスト用ディレクトリはありません。

## ビルド・テスト・開発コマンド
ビルドやスクリプトはありません。ツール（例: 静的サイトジェネレーター）を追加した場合は、ここに具体的なコマンドを記載してください。例:

- `npm run build`: LPアセットを`dist/`にビルド。
- `npm run dev`: ローカルのプレビューサーバーを起動。

## コーディングスタイルと命名規則
- 主な形式はMarkdown。見出しは簡潔にし、箇条書きの体裁を統一する。
- 既存の日本語ファイル名と`LP設計案/`の番号順（例: `01_...`, `02_...`）を維持する。
- 図表は、可能であれば編集可能な元データをエクスポート済みアセットと並べて置く（例: `.svg`と対応する元ファイル）。

## テスト方針
テストは定義されていません。コードを追加する場合は最小限のテスト環境を用意し、実行方法を記載してください。コンテンツ変更時は、Markdownの表示とSVGの描画を手動で確認してください。

## レンダリング確認チェックリスト
- 見出し階層が意図通り（H1 → H2 → H3）に表示されるか確認する。
- Markdownプレビューで箇条書きのインデントと余白を確認する。
- `LP設計案/03_再設計フロー.svg`を開き、文字が読みやすく欠けていないか確認する。

## コミットとプルリクエストの方針
この作業スペースにはGit履歴がないため、既存のコミット規約は要約できません。Gitを使う場合は以下を参考にしてください:

- コミットメッセージは短く命令形で、範囲を明確にする（例: `Update LP wireframe flow`）。
- プルリクエストには簡単な概要、関連Issue、視覚変更やSVG変更のスクリーンショットを含める。

## アセットとコンテンツの注意点
- 編集可能な元データがないバイナリアセットは直接編集しない。元データを更新して再エクスポートする。
- 変更は絞る。可能ならコミット/PRは1ドキュメントまたは1テーマに集中させる。
